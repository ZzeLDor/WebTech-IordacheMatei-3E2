1. Introducere

1.1 Scopul Documentului

Acest document detaliaza Specificatia Cerintelor de Sistem (SRS) pentru aplicatia **KIM Web Manager**. Documentul stabileste cerintele functionale, de performanta, interfata si securitate care ghideaza dezvoltarea aplicatiei dedicate gestiunii activitatilor sportive si de recuperare medicala (kinetoterapie).

1.2 Conventii de Documentare

Acest document respecta structura standard oferita de **IEEE Std 830-1998** pentru descrierea cerintelor software. Cerintele prioritare sunt marcate ca **Esentiale**, iar termenii tehnici folositi respecta arhitectura MVC implementata.

1.3 Publicul Tinta si Sugestii

Documentul se adreseaza evaluatorilor academici ai proiectului, echipei de dezvoltare frontend/backend si administratorilor de sistem care doresc sa inteleaga fluxul de rezervari si integrarea API din KIM.

1.4 Prezentarea Generala a Sistemului

**KIM Web Manager** este un sistem web modular ce faciliteaza programarea sedintelor de fitness si kinetoterapie, coordonarea echipei de specialisti (antrenori si terapeuti), urmarirea gradului de ocupare a salilor (resurselor) si validarea automata a tipurilor de abonamente pentru clienti. Comunicarea se face complet asincron, prin AJAX Fetch API.

2. Descriere Generala

2.1 Perspectiva Produsului

Sistemul functioneaza independent ca o platforma de gestiune interna a unei clinici de kinetoterapie/sali de fitness. Este organizat pe un model arhitectural MVC personalizat, scris in PHP nativ, cu stocare pe SQLite pentru simplitate si portabilitate maxima in mediul universitar.

2.2 Functiile Sistemului

Aplicatia furnizeaza servicii asincrone pentru urmatoarele fluxuri operationale majore:

  • Autentificare si inregistrare diferentiata pe baza de roluri.
  • Gestiunea orelor saptamanale de grup si individuale (Timetable).
  • Flux de cereri private de la clienti la terapeuti/antrenori (cu notificare prin e-mail).
  • Alocarea echipamentelor pe sali si raportarea starii acestora.
  • Dashboard de analiza manageriala cu grafice evolutive si descarcari in format CSV si XML.

2.3 Clase de Utilizatori si Caracteristici

Rol Utilizator Nivel Acces Drepturi Principale
Membru Scazut Vizualizare calendar, rezervare sedinte de grup permise de abonament, trimitere cereri antrenamente private.
Antrenor / Kinetoterapeut Mediu Editare orar propriu, programare sedinte de grup, aprobare/respingere cereri private primite de la clienti.
Administrator Maxim Modificare roluri/specializari, alocare echipamente, configurare sali, acces complet la statistici si export CSV/XML.

2.4 Mediul de Operare

Aplicatia ruleaza pe orice server web compatibil cu PHP (PHP 7.4 - 8.x) cu extensia `pdo_sqlite` activata. Interfata de utilizator este complet compatibila cu navigatoarele moderne (Chrome, Firefox, Edge, Safari), necesitand suport nativ pentru standardul ES6 Javascript.

2.5 Constrangeri de Design si Implementare

Pentru a asigura o curatenie absoluta a structurii academice si conform solicitarii clientului, codul sursa a fost curatat complet de comentarii inline in toate fisierele PHP, JS, CSS si SQL. Interfata foloseste exclusiv componente CSS native, fara biblioteci sau framework-uri externe (precum TailwindCSS).

2.6 Dependente si Ipoteze

Sistemul presupune existenta unei conexiuni functionale la internet pentru incarcarea setului de fonturi (Google Fonts) si a bibliotecii de diagrame/statistici (Chart.js), precum si pentru serviciul de expediere a notificarilor SMTP.

3. Cerinte Functionale Esentiale

3.1 Inregistrare si Autentificare Securizata

Sistemul trebuie sa permita utilizatorilor crearea unui cont prin furnizarea numelui, e-mailului unic si a parolei. Parola este hash-uita prin algoritmul BCRYPT. Controlul accesului este asigurat prin sesiuni backend securizate.

3.2 Validarea Automata a Abonamentelor

Regula de afaceri cruciala: Un membru poate rezerva o sedinta doar daca are un abonament **activ** (neexpirat) si care este compatibil cu specificul activitatii dorite.

Daca o sedinta este de categoria *Kinetoterapie*, utilizatorul trebuie sa aiba un abonament de tip `kineto`, `mixed`, `medical` sau `premium`. Pentru sedintele de tip *Forta*, este necesar un abonament `forta`, `strength`, `mixed`, `premium` sau `general`. Validarea are loc pe backend, in controllerul `SessionController.php` prin metoda `canBookCategory()`.

3.3 Orar Saptamanal Interactiv si Inscrieri

Platforma ofera o vizualizare interactiva de tip grila calendaristica (Timetable). Membrii se pot inscrie printr-un singur click pe sedintele de grup active, cu conditia ca limita de capacitate maxima a salii sa nu fie atinsa. Inscrierea salveaza automat o inregistrare in tabelul `bookings`.

3.4 Reguli de Evitare a Suprapunerilor (Sala Principala vs. Cabinete)

La crearea sau editarea unei sedinte in calendar, sistemul face verificari asincrone pentru a evita suprapunerile de timp. Regula de validare respecta urmatoarea specificitate:

  • Salile normale (ID diferit de 1): Au utilizare exclusiva. Nu pot fi ocupate de doua sedinte diferite in acelasi interval de timp.
  • Sala Principala (ID 1): Permite programari concurente din partea diferitilor antrenori (suprapunerea fiind considerata valida deoarece spatiul este proiectat pentru utilizare colectiva).

3.5 Gestionarea Cererilor Private si Notificari Mail

Utilizatorii pot trimite cereri directe de antrenament privat sau terapie catre anumiti specialisti din echipa. La initierea unei cereri, specialistul primeste o notificare prin e-mail realizata prin serviciul `Core\Mailer`. Specialistul are dreptul sa aprobe sau sa respinga cererea. Aprobarea converteste automat cererea intr-o sedinta de tip `(Privat)` in baza de date.

De asemenea, `Core\Mailer` este utilizat si in cadrul `SessionController` pentru a expedia e-mailuri automate de confirmare la rezervarea unei sedinte de catre membru, respectiv de notificare in cazul in care o sedinta este anulata de catre administrator sau de catre antrenor.

3.6 Panou Administrativ Complex si Export Date

Administratorul dispune de un dashboard cu vizualizari dinamice alimentate asincron pentru monitorizarea performantei. Graficele pot fi salvate ca imagini (`PNG`/`WebP`). De asemenea, sistemul exporta rapoarte statistice in formatele structurate standard solicitate in mediul academic:

  • Export CSV: Date tabelare simplificate, usor importabile in foi de calcul.
  • Export XML: Structura arborescenta ierarhizata a rapoartelor pentru interoperabilitate ridicata.

4. Cerinte de Interfata Externa

Interfata de interactiune cu utilizatorul foloseste concepte moderne de design, cum ar fi:

  • Design Responsiv: Adaptabilitate pe dispozitive mobile prin media-queries in `style.css`.
  • Sistem de Toast-uri asincrone: Notificari dinamice pe ecran (succes, eroare, atentionare) pentru feedback instant, fara a reincarca pagina.
  • Design Premium: Utilizarea fontului Outfit de la Google Fonts, palette de culori elegante bazate pe albastru regal si elemente vizuale atractive.

5. Cerinte Nefunctionale (Calitati ale Sistemului)

  • Securitate si Prevenire Atacuri: Parole securizate prin BCRYPT, validari de rol la nivel de controller, prevenirea SQL Injection prin PDO si protectie XSS prin htmlspecialchars() la afisare.
  • Performanta: Timp de raspuns al API-ului sub 150ms datorita optimizarii interogarilor SQLite si stocarii locale.
  • Portabilitate: Aplicatia functioneaza perfect indiferent de platforma de operare (Windows, Linux, macOS) atat timp cat PHP este disponibil.
  • Robustete: Prinderea corecta a erorilor de baza de date cu returnarea de mesaje de eroare intuitive in format JSON (`{status: "eroare", eroare: "detaliu"}`).
